# v0.8 Implementation Plan

Status: planned.

v0.7 made compiled almanacs answerable through `almanac ask`. v0.8 should make
that answer path inspectable, replayable, and gateable without expanding almanac
into a general-purpose personal memory runtime.

The planned theme is **answer quality and diagnostics**:

- structured planner/tool/citation/abstain traces in saved answer artifacts,
- human-readable `runs` detail output for debugging answer sessions,
- deterministic replay from saved answer runs or fixture JSONL,
- ask-mode quality gates that measure citation, freshness, and abstention
  behavior separately from compile-time benchmarks,
- `doctor` and `profile` readiness signals that explain whether an almanac is
  ready for answer-mode use.

## Goals

- Make every saved answer session explainable enough for an operator to debug
  why it answered, abstained, or failed.
- Preserve the v0.7 runtime boundary: the planner may only call enabled compiled
  tools through `AlmanacRuntime.execTool`.
- Add deterministic replay so answer-mode regressions can be caught without
  requiring a live LLM provider on every test run.
- Add quality gates for cited-answer behavior: citation coverage, unsupported
  claim count, stale citation handling, and expected abstention behavior.
- Surface answer readiness in `doctor` and `profile` without making those
  commands perform hidden LLM calls.
- Keep existing Stage 11/12 benchmark gates intact; ask-mode gates are additive,
  not a replacement.

## Non-goals

- No multi-turn chat memory.
- No hosted answer daemon or resident agent.
- No arbitrary web browsing outside compiled tool capabilities.
- No automatic refresh before answering.
- No weakening of benchmark coverage or citation gates to make answer checks
  pass.
- No graph runtime, skillpack runtime, personal memory layer, or always-on dream
  cycle in v0.8.
- No OAuth, user login, or hosted multi-tenant deployment surface.
- No full entailment proof in the first quality gate. v0.8 should provide
  deterministic operational gates; deeper LLM judging can be considered later.

## What We Learned From gbrain

The review of [`garrytan/gbrain`](https://github.com/garrytan/gbrain) is useful
because it shows what an agent-native knowledge runtime looks like when
diagnostics, evals, and operator runbooks are treated as product features.

Adopt these ideas:

- **Traceability as a first-class artifact.** gbrain's strongest operational
  pattern is that search, eval, doctor, and run history are debuggable. Almanac
  should apply that to answer sessions by preserving planner decisions, tool
  observations, citation selection, freshness warnings, and abstention reasons.
- **Replayable evaluation.** gbrain's replay/eval surfaces are valuable because
  they make retrieval changes measurable. Almanac should add ask replay fixtures
  so answer behavior can be regression-tested separately from generated
  compile-time benchmark fixtures.
- **Doctor/readiness as product UX.** Operators should not need to inspect JSON
  by hand to know whether answer mode is ready. `doctor` and `profile` should
  summarize missing fixtures, stale evidence, last answer failures, and missing
  quality gates.
- **Cost and provider boundaries.** LLM-backed commands should stay explicit,
  budgeted, and visible in saved artifacts.

Do not adopt these parts in v0.8:

- gbrain's broad personal/company brain scope,
- graph-first personal memory,
- skillpack/runtime agent identity,
- dream/autopilot cycles,
- hosted OAuth/multi-tenant server surface,
- very large CLI surface area.

Almanac's narrower job is to compile and operate domain-specialized knowledge
surfaces. v0.8 should deepen the answer path, not turn almanac into a general
memory operating system.

## Product Shape

The existing answer command remains the entry point:

```bash
almanac ask sqlite-demo "Are SQLite transactions atomic?" --json
almanac ask sqlite-demo "Are SQLite transactions atomic?" --save --label rc-answer
```

Saved answer artifacts should become the canonical debugging record:

```bash
almanac runs sqlite-demo --kind answer --latest
almanac runs sqlite-demo <answer-artifact-id>
almanac runs sqlite-demo <answer-artifact-id> --json
```

Replay command naming can be finalized in PR3. The intended contract is:

```bash
almanac ask-replay sqlite-demo --fixture fixtures/ask.jsonl --json
almanac ask-replay sqlite-demo --from-runs --label rc-answer --json
```

Replay must have a no-provider deterministic path. A release gate should be able
to replay recorded tool calls and run quality checks without making a new LLM
call.

## Answer Trace Contract

Answer artifacts already record enough summary information to prove that a run
happened. v0.8 should add a structured trace that explains the answer session.

Recommended trace fields:

- `trace.schemaVersion`
- `trace.plannerPromptVersion`
- `trace.synthesisPromptVersion`
- `trace.budgets`: max tool calls, max duration, max answer chars
- `trace.plannerSteps[]`: iteration number, selected tool, redacted input,
  validation status, stop reason, repair attempts, and model error if any
- `trace.toolObservations[]`: tool name, input hash or redacted input, status,
  duration, citation count, freshness summary, error code, and output size
- `trace.citationLedger[]`: citation id, source id, title/url when available,
  fetched/source timestamps, volatility, stale flag, producing tool call, and
  whether the final answer used it
- `trace.abstention`: reason, blocking condition, tool failures, and whether the
  abstention was expected by replay/gate fixtures
- `trace.quality`: citation rate, unsupported claim count, stale citation count,
  and quality verdict when a gate has been run

Secrets must remain excluded:

- no API keys,
- no raw provider request/response body by default,
- no environment dump,
- no full raw source documents beyond tool-returned citation snippets,
- no unredacted sensitive tool input.

## Ask Replay Fixtures

Replay should support two inputs.

### Saved Answer Artifacts

Saved artifacts can be replayed by re-executing recorded deterministic tool
calls, then re-running citation and quality gates against the recorded final
answer or abstention. This mode is no-provider by default.

Use cases:

- prove a saved answer still passes quality gates after gate logic changes,
- detect if a compiled tool's deterministic result changed,
- debug why a prior answer failed without calling the provider again.

### Fixture JSONL

Fixture rows should be small, reviewable, and stable:

```json
{"id":"sqlite-transactions-ok","question":"Are SQLite transactions atomic?","toolCalls":[{"tool":"query_facts","input":{"q":"transactions atomic"}}],"expectedStatus":"ok","minCitations":1,"maxUnsupportedClaims":0}
{"id":"sqlite-out-of-scope-abstain","question":"Who won the 2026 World Cup?","toolCalls":[{"tool":"query_facts","input":{"q":"2026 World Cup"}}],"expectedStatus":"abstained","expectedAbstentionReason":"no-citations"}
```

The exact schema should be finalized in PR3, but it should include:

- stable fixture id,
- question,
- recorded or declared tool calls,
- expected status,
- minimum citation count or citation rate,
- maximum unsupported claim count,
- stale citation expectation,
- expected abstention reason when applicable.

Replay should not replace Stage 11 benchmark generation. It is an answer-mode
regression tool for saved or hand-authored answer behavior.

## Answer Quality Gates

v0.7's cite-or-abstain gate is intentionally strict but minimal. v0.8 should add
an ask-mode gate that is still deterministic enough for CI.

Recommended gate inputs:

- answer artifact or replay fixture,
- final answer status and abstention reason,
- final citation list,
- structured claim map from synthesis,
- citation freshness metadata,
- tool observation statuses.

Recommended gate metrics:

- `citationRate`: cited claims divided by total claims, or 1.0 when the answer
  is intentionally citation-only and has no separate claim map.
- `unsupportedClaimCount`: claims without a citation from the observed
  citation ledger.
- `staleCitationCount`: final citations whose freshness metadata marks them
  stale.
- `staleCitationDisclosure`: whether stale citations are disclosed in the answer
  or structured warnings.
- `abstainExpectedActual`: fixture expected status/reason compared with observed
  status/reason.
- `toolErrorCount`: failed tool observations that influenced the answer.

Recommended default thresholds:

- `status: "ok"` requires at least one final citation.
- `unsupportedClaimCount` must be `0`.
- stale citations are allowed only when the answer explicitly treats the claim
  as historical or surfaces a freshness warning.
- expected abstentions must abstain with the expected reason family.

This gate is not a proof that every sentence is true. The synthesis step should
emit a structured claim map so the deterministic gate can enforce that each
claim is attached to observed citations. Future releases can add optional LLM
entailment judging on top.

## Doctor And Profile Readiness

`doctor` and `profile` should surface answer readiness without making provider
calls.

Potential readiness inputs:

- compiled tools exist and expose citable outputs,
- Stage 12 benchmark status,
- ask replay fixtures exist,
- latest saved answer artifact status,
- latest ask quality gate verdict,
- stale source/fact summary,
- missing provider key for real `ask` runs,
- missing or outdated answer trace schema,
- missing quality gate config.

Potential readiness states:

- `ready`: benchmark passed, replay fixtures exist, latest ask gate passed, no
  stale blocking issue.
- `needs-validation`: answer mode can run, but fixtures/gates are missing or
  stale evidence needs review.
- `not-ready`: benchmark missing/failed, no compiled tools, malformed answer
  artifacts, or latest answer failure indicates a runtime issue.

Example profile output:

```text
answer mode    needs-validation
ask fixtures   0 found
latest answer  abstained (no-citations), 2026-06-01T10:15:00Z
quality gate   missing
```

### PR4: Answer Quality Gates

Add ask-mode quality gates over answer artifacts and replay results.

Implementation:

- `src/manage/answer-quality.ts` evaluates citation coverage, stale citations,
  unsupported claim count, and abstention expectation matching.
- Live answer traces persist a compact `trace.quality` verdict.
- `almanac ask-replay` reports per-case quality and aggregate quality summary
  alongside replay pass/fail status.
- Fixture JSONL supports `unsupportedClaims` and `maxUnsupportedClaims` for
  deterministic regression checks without provider calls.

Acceptance:

- Gate reports citation rate, unsupported claim count, stale citation handling,
  and abstain expected/actual.
- `status: "ok"` answers with unsupported claims fail the gate.
- Stale citations fail unless disclosed as stale or historical.
- Expected abstention fixtures fail when the answer succeeds or abstains for the
  wrong reason family.
- Stage 11/12 benchmark behavior is unchanged.

Validation:

- focused quality gate tests
- replay fixture tests covering ok, unsupported claim, and abstention cases
- `bun run typecheck`
- `bun test`

### PR5: Doctor/Profile Answer Readiness

Expose ask-mode readiness in `doctor` and `profile`.

Implementation:

- `src/manage/answer-readiness.ts` reads fixture JSONL coverage and saved answer
  artifacts without calling providers.
- `profile` prints `answer mode`, `ask fixtures`, `latest answer`, and
  `quality gate` lines, with answer-specific readiness gaps separated from the
  existing almanac readiness status.
- `doctor` adds an `answer` check with actionable warnings for missing fixtures,
  missing answer artifacts, missing quality gates, latest failures, and stale
  citations.
- JSON profile output includes the full answer readiness object.

Acceptance:

- `profile` summarizes answer mode readiness, fixture coverage, latest answer
  status, latest quality gate status, and stale evidence warnings.
- `doctor` emits actionable warnings for missing ask fixtures, missing quality
  gate results, malformed answer artifacts, stale blocking evidence, and latest
  saved answer failures.
- Readiness checks do not call providers.
- Existing refresh and benchmark readiness output remains stable.

Validation:

- focused profile/doctor tests
- `bun run typecheck`
- `bun test`

## v0.8 RC Smoke

Use disposable roots where possible. The deterministic replay checks should not
require provider credentials; real-provider ask smoke should require
`ANTHROPIC_API_KEY`.

### sqlite-demo

```bash
tmp=$(mktemp -d)

bun src/cli.ts demo --root "$tmp"
bun src/cli.ts inspect sqlite-demo --root "$tmp"
bun src/cli.ts profile sqlite-demo --root "$tmp"
bun src/cli.ts benchmark sqlite-demo --root "$tmp"

bun src/cli.ts ask sqlite-demo \
  "Are SQLite transactions atomic?" \
  --save \
  --label rc-answer \
  --json \
  --root "$tmp"

bun src/cli.ts runs sqlite-demo --kind answer --latest --json --root "$tmp"
bun src/cli.ts runs sqlite-demo --kind answer --latest --root "$tmp"

bun src/cli.ts ask-replay sqlite-demo \
  --from-runs \
  --label rc-answer \
  --json \
  --root "$tmp"

bun src/cli.ts doctor sqlite-demo --root "$tmp"
```

RC success criteria:

- typecheck and full test suite pass,
- sqlite-demo benchmark passes,
- saved answer artifact includes trace fields,
- `runs` detail output is readable for answer artifacts,
- deterministic replay passes without provider keys,
- Enterprise AI benchmark remains at or above generated coverage floors,
- real-provider ask smoke returns either a cited answer or a correct abstention,
- `doctor` and `profile` expose answer readiness without hidden provider calls.
