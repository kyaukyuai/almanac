# v0.4 Implementation Plan

Status: completed for v0.4.0 on 2026-05-29.

v0.4 focused on measurable answer quality and inspectability. The sequence
started with benchmark coverage for comparison questions, then stabilized source
sets, then added hybrid retrieval, transport, and export features in separate
PRs. Each PR left `main` shippable.

Implemented as PRs #6 through #13.

## Goals

- Make comparison / tradeoff knowledge measurable in generated benchmarks.
- Reduce cross-run benchmark noise caused by source-discovery drift.
- Add vector retrieval without weakening the existing cite-or-abstain contract.
- Keep runtime transports and inspection exports independent from compile-time
  LLM behavior.

## Non-goals

- No hosted service or marketplace in v0.4.
- No local CLI LLM orchestrator (`almanac run` remains v0.5+).
- No replacement of FTS5; vector retrieval is additive and benchmark-gated.
- No broad source snapshot allowlist beyond explicitly verified permissive docs.

### PR2: Tradeoff-aware Stage 11 fixtures

Teach Stage 11 to generate at least one comparison-shaped positive fixture when
both conditions hold:

- the fact sample includes one or more `tradeoff` facts with at least two
  entities, and
- an enabled tool can answer comparison-shaped queries, either through
  fact-store search or an input schema with paired values such as `{a, b}`.

Implementation notes:

- Start prompt-first in `src/compile/prompts/11-benchmark-gen/v3.md`.
- Add deterministic pre-normalization only if prompt guidance is not enough.
- Add tests around `src/compile/stages/s11-benchmark-gen.test.ts`.
- Avoid forcing tradeoff fixtures when the corpus has no tradeoff facts.

Validation:

- `bun test src/compile/stages/s11-benchmark-gen.test.ts`
- `bun run typecheck`
- `bun test`

Acceptance:

- A seeded tradeoff fact sample yields a positive fixture whose invocation and
  expected shape are grounded in the paired entities.
- Existing no-tradeoff benchmark generation remains valid.

### PR3: Stabilize Approved Source Reuse

Reduce source-discovery non-determinism by reusing previously approved sources
when an update starts before or at Stage 02x and the existing sources are still
fetchable and in scope.

Implementation notes:

- Prefer a deterministic carry-forward step over LLM prompt reliance.
- Preserve human-approved source ids and notes.
- New sources may still be added when coverage goals are not met.
- Do not carry forward sources that are failed, out-of-scope, or explicitly
  rejected in the new candidate set.

Likely touch points:

- `src/compile/stages/s02b-source-discovery-evaluator.ts`
- `src/compile/stages/s03-approve-runner.ts`
- `src/core/types.ts` if reuse metadata needs to be represented

Validation:

- Stage 02b unit tests for carry-forward and rejection cases.
- `bun run typecheck`
- `bun test`

Acceptance:

- Re-running source discovery with equivalent candidates preserves accepted
  source ids and ingestion modes.
- Benchmark fixture comparisons become more meaningful across runs because the
  source set is less volatile.

### PR4: Embedding Provider Abstraction

Introduce an internal embedding interface without changing retrieval behavior.

Implementation notes:

- Add provider-neutral types for embedding requests, responses, dimensions, and
  model identity.
- Support at least one deterministic test provider.
- Add env/config discovery for future providers, but do not call external APIs
  in tests.
- Keep `ToolManifest.knowledgeUsage.embeddings` defaulting to `false`.

Likely touch points:

- `src/core/types.ts`
- new `src/embeddings/*` or `src/knowledge/embeddings/*`
- `src/cli.ts` doctor warnings for configured providers

Validation:

- Provider unit tests.
- `bun run typecheck`
- `bun test`

Acceptance:

- No runtime behavior changes.
- Later PRs can request embeddings through one stable interface.

### PR5: Vector Index Artifacts

Build and persist vector index artifacts during Stage 08 when embeddings are
enabled.

Implementation notes:

- Keep FTS5 index generation unchanged.
- Persist a manifest that records embedding provider, model, dimensions, source
  fact corpus hash, and artifact paths.
- Make vector indexing opt-in until PR6 proves retrieval quality.
- Skip gracefully when no provider credentials are configured.

Likely touch points:

- `src/compile/stages/s08-knowledge-index.ts`
- `src/core/types.ts`
- `src/cli.ts` inspect/profile/doctor summaries

Validation:

- Stage 08 tests with deterministic embeddings.
- `almanac demo` remains no-key and warning-free apart from expected env
  warnings.
- `bun run typecheck`
- `bun test`

Acceptance:

- Vector artifacts are reproducible from the same fact corpus and deterministic
  provider.
- Missing credentials do not break existing compile paths.

### PR6: Hybrid Retrieval With RRF

Add retrieval that fuses FTS5 BM25 and vector similarity with reciprocal rank
fusion.

Implementation notes:

- Start with RRF because it needs no per-domain weighting.
- Preserve citation behavior and freshness computation.
- Fall back to FTS-only when vector artifacts are absent.
- Add enough observability to tell whether a result came from FTS, vector, or
  both.

Likely touch points:

- `src/compile/stages/s08-knowledge-index.ts`
- `src/serve/runtime.ts`
- default `query_facts` template under `src/compile/stages/s07/templates.ts`
- profile/doctor summaries

Validation:

- Retrieval unit tests with controlled FTS/vector rankings.
- Existing benchmark suite.
- At least one real-domain smoke comparing FTS-only vs hybrid.

Acceptance:

- Hybrid retrieval never drops citations.
- FTS-only behavior is preserved when vectors are unavailable.
- Benchmark results justify enabling hybrid by default or keeping it opt-in.

### PR7: HTTP/SSE MCP Transport

Expose the existing runtime over HTTP/SSE in addition to stdio.

Implementation notes:

- Runtime only; no compile-time changes.
- Keep stdio as the default.
- Add CLI flags such as `almanac serve <id> --transport=http --port=...` only
  after checking MCP SDK support.
- Include graceful shutdown and health endpoint if supported by the SDK.

Likely touch points:

- `src/serve/mcp-server.ts`
- `src/cli.ts`
- MCP server tests

Validation:

- stdio tests stay green.
- HTTP/SSE tests use local in-process server.
- `bun run typecheck`
- `bun test`

Acceptance:

- Existing client registrations are unaffected.
- HTTP/SSE server exposes the same tools and resources as stdio.

### PR8: Wiki Inspection Export

Generate a human-readable inspection bundle for a compiled almanac.

Implementation notes:

- Export Markdown files, not a new runtime format.
- Include overview, source review, evidence distribution, benchmark summary,
  tool list, and readiness gaps.
- Keep generated files deterministic for diff review.
- Consider integrating with `almanac export --include-wiki` after the basic
  command works.

Likely touch points:

- new `src/manage/wiki-export.ts`
- `src/cli.ts`
- `docs/product-guide.md`

Validation:

- Snapshot-style tests on generated Markdown.
- `bun run typecheck`
- `bun test`

Acceptance:

- A human can inspect an almanac without opening JSON artifacts.
- The wiki export agrees with `inspect`, `profile`, `sources`, and
  `benchmark`.

## Success Metrics

- Generated benchmark sets include comparison coverage when the corpus contains
  tradeoff facts.
- Source-discovery reruns preserve approved source identity unless there is a
  concrete reason to change it.
- Hybrid retrieval improves or preserves positive benchmark pass rate without
  reducing negative fixture pass rate.
- `profile` remains `usable` for demo and target smoke almanacs.
- No increase in uncited positive benchmark results.
