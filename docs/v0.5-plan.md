# v0.5 Implementation Plan

Status: shipped as v0.5.0 on 2026-06-01.

v0.5 makes a compiled almanac directly usable from the CLI, without requiring
a downstream MCP client to prove that the runtime works. The release preserves
the v0.4 compile pipeline and focuses on a narrow product layer:
local invocation, scriptable output, and enough run metadata to make later
refresh automation practical.

The shipped theme is **run-first operations**:

- direct tool execution for deterministic checks,
- inspection/export schema polish where it supports CI and handoff,
- refresh scheduling design, but no daemon or hosted service yet.

## Goals

- Add a first-class `almanac run` path for invoking compiled almanac tools from
  the local CLI.
- Make local runtime behavior easy to smoke-test in CI and demos, including
  stable JSON output and citation visibility.
- Reuse the existing `AlmanacRuntime` and `ToolResult` envelope instead of
  creating a separate execution stack.
- Keep no-key demo behavior intact.
- Keep refresh/scheduling and LLM-backed question mode out of the critical path
  until deterministic local runtime operations are stable.

## Non-goals

- No hosted service, marketplace, or private deployment product in v0.5.
- No Slack adapter in v0.5.
- No long-running scheduler daemon.
- No mandatory paid LLM call for basic `almanac run`; deterministic tool mode
  comes first.
- No weakening of the cite-or-abstain contract. CLI answers must expose the
  same citations and error envelopes as MCP execution.

## Product Shape

The desired CLI surface is:

```bash
almanac run <id> --tool query_facts --input '{"q":"foreign keys"}'
almanac run <id> --tool query_facts --input-file query.json --json
almanac run <id> --list-tools
```

Saved run artifacts are opt-in:

```bash
almanac run <id> --tool query_facts --input '{"q":"foreign keys"}' --save --label smoke
almanac runs <id> --label smoke --json
almanac runs <id> --prune --keep-latest 5 --dry-run
almanac export <id> --include-runs
```

LLM-backed question mode is deferred. It should be explicit about provider and
cost behavior when it is designed, and tests must use mock providers.

### PR4: Run Artifacts And Audit Trail

Persist optional run reports for CLI executions so debugging and later
scheduling work have a durable record.

Shipped shape:

- Add opt-in `almanac run --save`; avoid writing files by default.
- Store reports under `.runs/`.
- Include almanac id, version, tool name, input, output status, result
  envelope, citation count, elapsed time, and timestamp.
- Do not store secrets or raw environment values.

### PR6: Run Artifact Metadata

### PR7: Run Artifact Selection/Filters

### PR8: Run Artifact Retention/Cleanup

Add safe cleanup for saved run artifacts.

Shipped shape:

- `almanac runs --prune`.
- Retention criteria through `--keep-latest` and `--older-than`.

Shipped shape:

- `almanac export --include-runs` includes `.runs/`.

## Resolved Decisions

1. **Command semantics.** v0.5 is direct tool mode only: `--tool` invokes one
   compiled tool and no hidden LLM call is made.
2. **Report location.** Saved run artifacts live in `.runs/`, outside
   `.compile/`, because they are operational audit records rather than compile
   intermediates.
3. **Artifact persistence.** `almanac run` does not write by default.
   Persistence requires `--save`; label/note metadata is allowed only with
   `--save`.
4. **Export behavior.** `.runs/` is excluded by default and included only with
   `--include-runs`.

## Open Decisions

1. **LLM provider default for future question mode.** Anthropic is already used
   for compile-time stages, but local question mode should be explicit.
   Recommendation: require `--provider` or environment configuration.
2. **Auto-refresh timing.** Scheduling is central to the product promise, but a
   daemon is too much for v0.5. Recommendation: implement in v0.6 after run
   reports and invocation semantics settle.
