# v0.5 Implementation Plan

Status: proposed on 2026-05-31.

v0.5 should make a compiled almanac directly usable from the CLI, without
requiring a downstream MCP client to prove that the runtime works. The release
should preserve the v0.4 compile pipeline and focus on a narrow product layer:
local invocation, scriptable output, and enough run metadata to make later
refresh automation practical.

The recommended theme is **run-first operations**:

- direct tool execution for deterministic checks,
- optional LLM-backed question mode after the deterministic runner is stable,
- inspection/export schema polish where it supports CI and handoff,
- refresh scheduling design, but no daemon or hosted service yet.

## Goals

- Add a first-class `almanac run` path for invoking compiled almanac tools from
  the local CLI.
- Make local runtime behavior easy to smoke-test in CI and demos, including
  stable JSON output and citation visibility.
- Reuse the existing `AlmanacRuntime` and `ToolResult` envelope instead of
  creating a separate execution stack.
- Keep no-key demo behavior intact.
- Define the refresh/scheduling contract needed for v0.6+ without committing to
  a background daemon in v0.5.

## Non-goals

- No hosted service, marketplace, or private deployment product in v0.5.
- No Slack adapter in v0.5.
- No long-running scheduler daemon.
- No mandatory paid LLM call for basic `almanac run`; deterministic tool mode
  comes first.
- No weakening of the cite-or-abstain contract. CLI answers must expose the
  same citations and error envelopes as MCP execution.

## Product Shape

The desired CLI surface is:

```bash
almanac run <id> --tool query_facts --input '{"q":"foreign keys"}'
almanac run <id> --tool query_facts --input-file query.json --json
almanac run <id> --list-tools
```

If deterministic tool mode lands cleanly, add an opt-in question mode:

```bash
almanac run <id> "What does SQLite WAL mode change?"
```

Question mode should be explicit about its provider and cost behavior. It may
use existing LLM provider plumbing, but tests must use mock providers and must
not require network credentials.

### PR2: Runtime Invocation Boundary

Factor a small management-layer helper that resolves an almanac directory,
loads `AlmanacRuntime`, invokes one tool, and returns the validated
`ToolResult` envelope.

Implementation notes:

- Keep the helper independent from Commander.
- Accept structured JSON input as an object, not an arbitrary string that tools
  parse themselves.
- Preserve runtime error codes and nonzero CLI exit mapping.
- Include a lightweight formatter for human output and a raw JSON path for CI.

Likely touch points:

- new `src/manage/run-tool.ts`
- `src/serve/runtime.ts`
- `src/core/runtime.ts`
- focused tests under `src/manage/`

Validation:

- `bun test src/manage/run-tool.test.ts`
- `bun run typecheck`
- `bun test`

Acceptance:

- A test almanac can invoke `query_facts` through the helper and receive
  citations.
- Missing tool, malformed input, and `ok:false` tool envelopes produce stable
  errors.

### PR3: `almanac run --tool`

Expose deterministic tool invocation in the CLI.

Implementation notes:

- Add `almanac run <id>` with `--tool`, `--input`, `--input-file`, `--json`,
  and `--root`.
- Add `--list-tools` so users can discover available tools without opening
  generated JSON.
- Default human output should show tool name, ok/error status, key result data,
  citations, and freshness.
- `--json` should print the exact validated envelope plus minimal command
  metadata.

Likely touch points:

- `src/cli.ts`
- `src/manage/run-tool.ts`
- CLI tests
- `README.md`
- `docs/product-guide.md`

Validation:

- CLI tests for list, successful tool invocation, bad input, and missing tool.
- `almanac demo` then `almanac run sqlite-demo --tool query_facts ...`.
- `bun run typecheck`
- `bun test`

Acceptance:

- A user can validate a compiled almanac without registering it with Claude
  Code, Claude Desktop, Cursor, Codex, or another MCP client.
- CI can assert result JSON, citation count, and freshness from one command.

### PR4: Run Artifacts And Audit Trail

Persist optional run reports for CLI executions so debugging and later
scheduling work have a durable record.

Implementation notes:

- Add an opt-in `--save-report` flag first; avoid writing files by default.
- Store reports under `.run/` or `.compile/runs/` only after choosing the
  ownership boundary.
- Include almanac id, version, tool name, input hash, output status, citation
  count, freshness, elapsed time, and timestamp.
- Do not store secrets or raw environment values.

Validation:

- Unit tests for report shape and redaction.
- Demo run with `--save-report`.
- `bun run typecheck`
- `bun test`

Acceptance:

- Saved reports are stable enough for bug reports and future scheduled-run
  comparisons.
- Reports do not perturb normal runtime behavior.

### PR5: Optional LLM Question Mode

Add natural-language `almanac run <id> "<question>"` only after deterministic
tool invocation is stable.

Implementation notes:

- Use a small local orchestration loop over the compiled tool manifests.
- Require explicit provider configuration; no hidden network calls.
- Prefer one or two tool calls before answering. Keep max turns configurable and
  low by default.
- Preserve citations in the final answer and expose the tool transcript with
  `--json` or `--trace`.
- Tests must use a mock LLM provider.

Validation:

- Mock-provider tests for tool selection, citation propagation, refusal when no
  source exists, and provider-missing errors.
- No-key demo remains unaffected.
- `bun run typecheck`
- `bun test`

Acceptance:

- Question mode can answer a demo SQLite question using `query_facts`.
- If no cited tool result is available, the CLI abstains instead of inventing an
  answer.

### PR6: Export And Wiki Schema Polish

Make inspection artifacts easier to consume from external tools and CI.

Implementation notes:

- Document `wiki/artifacts.json` fields and schema version.
- Consider `almanac export --include-wiki` once the standalone wiki bundle is
  stable.
- Keep `artifacts.json.files` complete, including the self-entry added in
  v0.4.2.

Validation:

- Wiki export tests.
- Export tests if `--include-wiki` is added.
- Fresh demo/wiki smoke.
- `bun run typecheck`
- `bun test`

Acceptance:

- A downstream process can validate wiki/export contents without scraping CLI
  text.

### PR7: Refresh Scheduling Design

Write the v0.6-ready scheduling contract without shipping a daemon.

Implementation notes:

- Define what a scheduled refresh would run: source fetch, fact extraction,
  index rebuild, benchmark gate, or full compile.
- Define lock behavior, retry behavior, stale-run detection, and notification
  surfaces.
- Decide whether run reports from PR4 are sufficient as the future scheduler's
  execution record.
- Keep the implementation to docs and maybe metadata types unless there is a
  clear low-risk CLI preflight command.

Validation:

- `git diff --check`
- Type/schema tests only if metadata types are added.

Acceptance:

- v0.6 can start implementation without reopening core product questions about
  ownership, locking, failure modes, or report shape.

## Open Decisions

1. **Command semantics.** Should `almanac run <id>` default to direct tool mode
   and require `--ask` for LLM mode, or should a positional question imply LLM
   mode? Recommendation: direct mode when `--tool` is present; question mode
   only when a question argument is present.
2. **Report location.** `.run/` is clearer to users; `.compile/runs/` keeps
   generated operational artifacts under the compile namespace. Recommendation:
   start with `.compile/runs/` because reports are derived artifacts.
3. **LLM provider default.** Anthropic is already used for compile-time stages,
   but local question mode should be explicit. Recommendation: require
   `--provider` or environment configuration in v0.5.
4. **Auto-refresh timing.** Scheduling is central to the product promise, but a
   daemon is too much for v0.5. Recommendation: design in v0.5, implement in
   v0.6 after run reports and invocation semantics settle.
