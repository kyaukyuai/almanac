# v0.9 Implementation Plan

Status: planned.

v0.8 made answer mode inspectable, replayable, and gateable. It added
structured answer traces, deterministic saved-answer replay, answer quality
gates, and doctor/profile readiness signals. v0.9 should turn those primitives
into a maintained ask-mode regression suite that can run during release checks
and after refreshes.

The planned theme is **ask suite operations**:

- author and maintain answer replay fixtures,
- run ask-mode suites with stable pass/fail output and exit codes,
- connect refresh runs to answer replay validation,
- make doctor/profile readiness actionable when ask fixtures or gate results
  are missing,
- keep answer verification deterministic unless an operator explicitly runs a
  live provider ask smoke.

## Goals

- Add a first-class fixture authoring workflow for `ask-replay`.
- Allow saved answer artifacts to be promoted into reviewable fixture JSONL.
- Add a suite-level CLI gate that can run all ask fixtures for an almanac.
- Make ask suite output suitable for CI, release smoke, and refresh follow-up.
- Integrate refresh workflows with optional post-refresh ask validation.
- Improve doctor/profile next actions so answer readiness can move from
  `needs-validation` to `ready` when fixtures and quality gates exist.
- Preserve the v0.8 runtime boundary: replay invokes compiled tools only and
  does not call an LLM provider.

## Non-goals

- No hosted scheduler or resident daemon in v0.9.
- No multi-turn chat memory.
- No personal-memory graph or broad agent runtime.
- No arbitrary web browsing outside compiled tool capabilities.
- No mandatory LLM judge or entailment model.
- No automatic refresh before answering.
- No weakening of Stage 11/12 benchmark gates.
- No hidden provider calls from `doctor`, `profile`, or replay commands.

## Lessons From v0.8

v0.8 shipped the right primitives, but the release smoke exposed the next
operational gap:

- `ask-replay` can validate saved runs and fixture JSONL, but there is no
  guided way to create or maintain fixture files.
- `doctor` and `profile` correctly warn about `no ask replay fixtures`; v0.9
  should make that warning easy to resolve.
- Release smoke can prove answer behavior manually, but ask-mode regression is
  not yet a standard suite gate alongside benchmarks.
- Refresh readiness and answer readiness are still separate surfaces; operators
  need a clean pattern for "refresh changed evidence, then replay answer
  fixtures."
- `ALMANAC_LLM=mock` is useful, but deterministic ask smoke requires explicit
  `ALMANAC_MOCK_RESPONSES`; v0.9 should avoid relying on ad hoc environment
  setup for normal regression checks.

## Product Shape

The existing replay command remains the deterministic engine:

```bash
almanac ask-replay sqlite-demo --fixture tests/ask.jsonl --json
almanac ask-replay sqlite-demo --from-runs --label rc-answer --json
```

v0.9 should add fixture authoring and suite-oriented commands without changing
the replay contract. Candidate shapes:

```bash
almanac ask-fixtures init sqlite-demo --root "$tmp"
almanac ask-fixtures add-from-run sqlite-demo <answer-id> --root "$tmp"
almanac ask-suite sqlite-demo --json --root "$tmp"
```

Exact naming can be finalized in PR2 and PR3. The stable product contract is:

1. fixtures are small JSONL files under the compiled almanac,
2. suite runs are deterministic and provider-free,
3. suite failures have actionable fixture ids and reason codes,
4. doctor/profile can tell the operator what is missing and what to run next.

## Fixture Authoring

v0.8 accepts fixture JSONL, but humans must write it by hand. v0.9 should add
a lightweight authoring workflow.

Recommended fixture locations, in priority order:

- `tests/ask.jsonl`
- `tests/ask-replay.jsonl`
- `fixtures/ask.jsonl`
- `fixtures/ask-replay.jsonl`

Starter fixture creation should support:

- a minimal positive fixture from a saved answer artifact,
- a minimal abstention fixture from a saved abstained answer,
- a hand-authored starter fixture using a declared `query_facts` tool call,
- validation that fixture ids are stable and unique.

Do not auto-generate a large fixture corpus with an LLM in v0.9. Fixture
authoring should be reviewable and deterministic.

## Ask Suite Gate

`ask-replay` currently reports one replay batch. v0.9 should add a suite-level
gate with clearer release semantics.

Recommended metrics:

- total fixtures, passed, failed, errored,
- status distribution: ok, abstained, model-error, tool-error,
- aggregate citation rate,
- unsupported claim count,
- stale citation count,
- expected/actual abstention mismatches,
- per-tool deterministic drift from saved expectations.

Recommended exit behavior:

- exit `0` when every fixture passes and aggregate quality passes,
- exit `1` when any fixture fails or aggregate quality fails,
- exit `2` for CLI usage or unreadable fixture errors.

The suite gate should remain separate from Stage 11/12 benchmarks. Benchmarks
validate generated runtime tool behavior; ask suites validate answer-mode
regressions over chosen questions.

## Refresh + Ask Validation

v0.6 made refreshes auditable, and v0.8 made answer replay deterministic. v0.9
should connect those capabilities.

Candidate workflow:

```bash
almanac refresh run enterprise-ai --save --label nightly --root "$ALMANAC_ROOT"
almanac ask-suite enterprise-ai --label post-refresh --json --root "$ALMANAC_ROOT"
```

Possible integration flags:

```bash
almanac refresh run enterprise-ai \
  --save \
  --label nightly \
  --ask-suite \
  --root "$ALMANAC_ROOT"
```

The first implementation can keep this as two explicit commands. If an
integration flag is added, the refresh artifact should record the ask-suite
summary without embedding raw answer artifacts or provider responses.

## Doctor And Profile Hardening

v0.8 already surfaces answer readiness. v0.9 should make it more actionable:

- show the discovered fixture file paths,
- show fixture count and latest suite result,
- distinguish "no fixtures" from "fixtures exist but have not been replayed",
- suggest exact commands to initialize fixtures or run the suite,
- make `ready` possible when benchmark passed, fixtures exist, latest suite
  passed, and latest saved answer quality passed.

Readiness checks must remain read-only and provider-free.

## v0.9 RC Smoke Target

The release candidate should pass:

- `bun run typecheck`,
- `bun test`,
- sqlite-demo fixture init/add-from-run,
- sqlite-demo ask suite pass,
- sqlite-demo refresh run followed by ask suite pass,
- Enterprise AI benchmark at or above generated coverage floors,
- Enterprise AI ask suite pass if fixtures exist,
- real-provider ask smoke returning either a cited answer or a correct
  abstention,
- saved answer artifact replay pass.

## Open Questions

1. Should the suite command be named `ask-suite`, `ask test`, or
   `ask-replay --suite`?
2. Should fixture authoring live under `ask-fixtures`, `ask-replay --init`, or
   `benchmark --ask-init`?
3. Should refresh artifacts link to ask suite artifacts directly, or only record
   a compact summary?
4. What is the minimum fixture count for answer mode to be considered ready?
5. Should v0.9 include an optional LLM entailment judge, or defer it until the
   deterministic gate has a fixture corpus in real use?
