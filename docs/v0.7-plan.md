# v0.7 Implementation Plan

Status: release-candidate documentation pass on 2026-06-01.

v0.7 should make a compiled almanac answerable from the CLI without weakening
the existing MCP-first architecture. v0.5 made individual tools runnable.
v0.6 made refreshes schedulable and auditable. v0.7 should add a thin,
explicit answer orchestration boundary over the already-compiled tool surface.

The theme is **cited answer mode**:

- a one-shot CLI command for asking a compiled almanac a question,
- LLM-backed tool selection over the compiled runtime, not arbitrary browsing,
- a final cite-or-abstain gate that requires grounded tool citations,
- optional saved answer artifacts under the existing `.runs/` audit surface.

This is not a hosted agent. It is a local operator and release-gate workflow
that proves an almanac can answer useful questions before a user registers it
with Claude Code, Claude Desktop, Cursor, Codex, or another MCP client.

## Goals

- Add a first-class answer session over a compiled almanac.
- Invoke only enabled compiled tools through `AlmanacRuntime.execTool`.
- Keep tool-call budgets, provider cost, and failure modes explicit.
- Preserve the existing cite-or-abstain contract: no grounded citation means no
  answer.
- Produce stable JSON suitable for release gates and CI.
- Optionally persist answer sessions as operational artifacts without storing
  provider secrets or environment dumps.
- Keep `almanac demo`, `almanac run --tool`, `almanac refresh due`, and
  read-only inspection paths no-key friendly.

## Non-goals

- No multi-turn chat memory in v0.7.
- No hosted scheduler, resident daemon, Slack adapter, marketplace, or private
  deployment product.
- No arbitrary web browsing outside compiled tool network capabilities.
- No automatic refresh before answering.
- No bypass around benchmark coverage gates.
- No hidden LLM call in `run --tool`, `runs`, `refresh due`, `inspect`,
  `profile`, `doctor`, `demo`, or `export`.
- No replacement for MCP clients. `ask` is a local verification and automation
  path, not the primary end-user chat surface.

## Product Shape

CLI:

```bash
almanac ask sqlite-demo "How do SQLite transactions behave?" --json
almanac ask sqlite-demo "When should I run ANALYZE?" --save --label rc-answer
almanac ask enterprise-ai "What governance checks should precede rollout?" --json
```

The command name is `ask`, and the contract is stable:

1. The question is answered only through compiled almanac tools.
2. Tool invocations are schema-validated and budgeted.
3. The final answer must cite tool-returned citations.
4. The result can be saved and inspected like other `.runs/` artifacts.

Human output should be compact:

```text
answer: sqlite-demo
status: ok
tools: query_facts
citations: 2

SQLite transactions are atomic ...

citations:
- sqlite-transactions — SQLite transaction documentation
```

JSON output should include enough structure for automation:

- `status`,
- `answer`,
- `abstentionReason`,
- `toolCalls`,
- `citations`,
- `freshness`,
- `durationMs`,
- `model`,
- `usage`,
- `savedArtifact` when `--save` is used.

## Answer Session Contract

An answer session should have three deterministic boundaries around the LLM:

1. **Context assembly.** Load manifest identity, domain contract files, enabled
   tool manifests, and optionally lightweight runtime resource descriptors.
2. **Tool planning and execution.** Ask the LLM for one tool call at a time,
   validate the tool name and input schema, execute through
   `AlmanacRuntime.execTool`, and repeat until the model chooses to stop or the
   budget is exhausted.
3. **Synthesis and gate.** Ask the LLM to synthesize from the observed tool
   envelopes, then validate that the final answer includes citations from the
   observed envelopes. If not, return `status: "abstained"`.

The session should never hand the LLM raw filesystem access or arbitrary
network access. All external effects are mediated by compiled tool manifests
and the runtime capability checks already used by MCP and `run --tool`.

Recommended default budgets:

- `maxToolCalls`: 4
- `maxDurationMs`: 120000
- `maxAnswerChars`: 6000
- no implicit retries beyond schema repair for malformed planner/synthesis
  JSON

Recommended statuses:

- `ok`
- `abstained`
- `tool-error`
- `tool-not-found`
- `bad-tool-input`
- `budget-exhausted`
- `model-error`

## Cite-Or-Abstain Gate

The final answer gate should be simple and strict in v0.7:

- `status: "ok"` requires at least one citation in the final answer.
- Every final citation must come from an observed tool result.
- If a cited tool result has stale or missing freshness metadata, the final
  output must expose that rather than silently treating it as current.
- If all tool calls return no citable evidence, the answer must abstain with a
  reason such as `no-citations`, `out-of-scope`, or `tool-errors-only`.

This gate is not a full entailment proof. It is an operational contract that
prevents the CLI from producing uncited prose when the compiled almanac could
not ground the answer.

## Operational Artifacts

v0.7 should extend the `.runs/` operational surface with answer artifacts.

Recommendation:

- Add a third artifact kind such as `answer`.
- Store saved sessions under `.runs/answer-*.json`.
- Keep `--save` explicit; unsaved answer sessions should not write artifacts.
- Reuse `almanac runs <id>` for listing, filtering, detail reads, retention,
  and export inclusion.

Answer artifact fields should include:

- artifact id, schema version, kind, almanac id, almanac version, forger
  version,
- question, label, note,
- model and prompt versions,
- started/finished timestamps and duration,
- status and exit code,
- normalized final answer or abstention reason,
- tool-call summaries with tool names, inputs, statuses, citation counts,
  durations, and error codes,
- final citations,
- usage metadata when the provider returns it.

Do not store:

- API keys,
- environment dumps,
- raw provider request/response bodies by default,
- unredacted secrets from tool input,
- full fetched documents beyond the citations already returned by tools.

## Provider And Cost Boundary

Answer mode is the first planned user-facing CLI command that intentionally
uses an LLM after compile time. It needs an explicit provider contract.

Recommended behavior:

- Default to the existing Anthropic provider configuration.
- Require `ANTHROPIC_API_KEY` unless a mock/test provider is injected.
- Print a clear error when credentials are absent.
- Include model and token usage in JSON and saved artifacts when available.
- Support `--model` only if the existing provider layer can do so without
  widening scope.

No-key workflows remain no-key because answer mode is opt-in:

- `almanac demo`
- `almanac run --tool`
- `almanac refresh due`
- `almanac inspect/profile/doctor`
- `almanac benchmark`

## Release Smoke Commands

Use a fresh disposable root. The grounded and abstention `ask` checks require a
real provider key unless the operator intentionally swaps in
`ALMANAC_LLM=mock` for a no-token CLI smoke.

```bash
tmp=$(mktemp -d)

bun src/cli.ts demo --root "$tmp"
bun src/cli.ts inspect sqlite-demo --root "$tmp"
bun src/cli.ts profile sqlite-demo --root "$tmp"
bun src/cli.ts benchmark sqlite-demo --root "$tmp"

bun src/cli.ts run sqlite-demo \
  --tool query_facts \
  --input '{"q":"transactions atomic"}' \
  --json \
  --root "$tmp"

test -n "$ANTHROPIC_API_KEY" || {
  echo "ANTHROPIC_API_KEY is required for real almanac ask smoke"
  exit 1
}

bun src/cli.ts ask sqlite-demo \
  "Are SQLite transactions atomic?" \
  --json \
  --root "$tmp"

bun src/cli.ts ask sqlite-demo \
  "What is the capital of France?" \
  --json \
  --root "$tmp" || true

bun src/cli.ts ask sqlite-demo \
  "Are SQLite transactions atomic?" \
  --save \
  --label rc-answer \
  --json \
  --root "$tmp"

bun src/cli.ts runs sqlite-demo --kind answer --json --root "$tmp"
bun src/cli.ts runs sqlite-demo \
  --kind answer \
  --prune \
  --keep-latest 1 \
  --dry-run \
  --json \
  --root "$tmp"
```

## Resolved Decisions

1. **v0.7 focus.** Implement answer mode before a resident scheduler. v0.6
   already gives cron/CI/launchd a stable refresh contract; answer mode closes
   the local usability gap.
2. **Interaction model.** v0.7 answer mode is one-shot, not multi-turn chat.
3. **Execution boundary.** The model can only call compiled almanac tools
   through the runtime.
4. **Grounding rule.** The final answer must cite observed tool citations or
   abstain.
5. **Audit location.** Saved answer sessions belong under `.runs/`, with
   explicit export inclusion.
6. **Command name.** The shipped CLI command is `ask`.
7. **Exit codes.** Grounded `ok` answers exit `0`; abstention and model/tool
   runtime failures exit `1`; usage and bad tool-input errors exit `2`.
   Automation should still inspect JSON `status` for detailed branching.
8. **Transcript retention.** Saved artifacts store normalized summaries,
   citations, usage, and final answer/abstention fields, not raw provider
   request/response bodies.
9. **Model override surface.** v0.7 exposes `--model` for answer planning and
   synthesis while keeping provider credentials environment-driven.
10. **Planner loop shape.** Planner and synthesis are separate prompts from
    the start.
11. **Answer evaluation.** Acceptance lives in focused CLI/session tests plus
    the release smoke commands above, not in Stage 11 benchmark fixtures.
