# v0.6 Implementation Plan

Status: proposed on 2026-06-01.

v0.6 turns the v0.5 local run layer into the foundation for scheduled refresh
operations. The product goal is not a hosted daemon yet. The goal is a
scriptable, auditable refresh contract that can tell operators what is due,
run the existing update pipeline safely, and leave durable evidence about what
happened.

The planned theme is **refresh scheduling readiness**:

- deterministic due detection before any write happens,
- explicit refresh execution over the existing compile/update pipeline,
- lock, retry, and failure semantics that are safe for CI and cron,
- operational artifacts that build on the v0.5 `.runs/` audit surface.

This plan tracks the v0.6 planning issue:
[`#34`](https://github.com/kyaukyuai/almanac/issues/34).

## Goals

- Define a first-class refresh unit for compiled almanacs.
- Add read-only due detection from source refresh metadata, compile state,
  benchmark state, and recent operational runs.
- Add a manual CLI refresh path that can be used by CI, cron, launchd, or a
  future daemon without changing semantics.
- Preserve the existing cite-or-abstain runtime contract and benchmark gate.
- Persist refresh attempts as auditable operational artifacts without storing
  secrets or raw environment values.
- Keep the no-key demo and deterministic `almanac run --tool` path intact.

## Non-goals

- No hosted scheduler, marketplace, Slack adapter, or private deployment in
  v0.6.
- No always-on daemon as the first implementation.
- No hidden LLM call in local tool invocation or read-only refresh planning.
- No weakening of benchmark coverage gates to make refreshes look successful.
- No automatic destructive cleanup of `.runs/` or source artifacts.

## Product Shape

The desired CLI surface is:

```bash
almanac refresh due <id> --json
almanac refresh run <id> --from-stage 04-source-fetch --json
almanac refresh run <id> --from-stage 04-source-fetch --save --label nightly
almanac refresh history <id> --latest 5 --json
```

The first release should be CLI-first and scheduler-friendly. A user should be
able to wire it into cron or CI without needing a resident process:

```bash
almanac refresh due enterprise-ai --json
almanac refresh run enterprise-ai --save --label cron-nightly
```

The command names are candidates. The important contract is:

1. `due` is read-only and deterministic.
2. `run` reuses the existing `update` pipeline.
3. refresh reports are inspectable and export behavior is explicit.

## Refresh Unit

The default refresh unit should be one almanac rooted at an existing almanac
directory. A refresh does not create a new almanac. It updates an existing one.

Default stage boundary:

- Start at `04-source-fetch` for routine source refreshes.
- Allow `--from-stage` for explicit operator control.
- Keep `02b-source-discovery-evaluator` available for source-set refreshes,
  but do not make it the default because it is LLM-heavy and more drift-prone.

Inputs for due detection:

- `sources/sources.json` ingestion mode and `refreshIntervalHours`.
- `sources/manifest.summary.json` fetch timestamps and source timestamps.
- `.compile/compile-state.json` stage status and last compile timestamp.
- `.compile/benchmark-result.json` pass/fail, citation rate, and fixture
  coverage.
- latest relevant `.runs/` refresh artifact status, when present.

Outputs:

- due/not-due status,
- reason codes such as `source-expired`, `stage-failed`,
  `benchmark-missing`, `benchmark-failed`, `never-refreshed`, and
  `manual-from-stage`,
- recommended `fromStage`,
- source ids that triggered the due state,
- enough timestamps for CI to assert behavior.

## Operational Artifacts

v0.5 introduced validated `.runs/run-*.json` artifacts for direct tool
execution. v0.6 should extend this operational audit surface carefully.

Recommendation:

- Introduce a `RunArtifactEnvelope` discriminated union.
- Keep existing tool artifacts backward compatible as `kind: "tool"` or by
  accepting the current v0.5 shape during reads.
- Add refresh artifacts as `kind: "refresh"`.
- Store refresh reports under `.runs/` so retention and export behavior remain
  one concept for operational records.
- Require explicit persistence at first, mirroring `almanac run --save`.

Refresh artifact fields should include:

- artifact id, schema version, kind, almanac id, almanac version, forger
  version,
- started/finished timestamps and duration,
- requested `fromStage` and effective `fromStage`,
- due decision snapshot,
- stage summary and final health summary,
- benchmark summary after the run,
- error code/message when failed,
- label/note metadata when explicitly provided.

Do not store:

- API keys,
- environment dumps,
- raw provider request/response bodies,
- unredacted secrets from tool input.

## Locking, Retry, And Failure

Refresh execution needs a small concurrency contract before any scheduler is
added.

Recommended behavior:

- Acquire a per-almanac lock before mutating stages.
- Fail fast with a stable `locked` error unless `--wait-lock` is provided.
- Record the existing lock holder pid, command, and timestamp when available.
- Treat stale lock cleanup as explicit operator action, not automatic default.
- Return nonzero when the pipeline fails, but still write a failed refresh
  artifact when `--save` is set and the artifact writer can run.

Retry policy:

- Do not retry the whole pipeline implicitly by default.
- Allow callers to retry externally using CI/cron.
- If an internal retry is later added, make attempt count visible in the
  refresh artifact and final CLI output.

## Resolved Decisions

1. **Implementation style.** v0.6 starts CLI-first. A daemon can be built later
   on top of the same contract.
2. **Default stage boundary.** Routine refresh starts at `04-source-fetch`.
   Earlier source discovery remains explicit operator intent.
3. **Read-only planning.** Due detection must not require LLM or search
   provider credentials.
4. **Audit location.** Refresh reports should live with operational run
   artifacts under `.runs/`, with explicit export inclusion.

## Open Decisions

1. **Artifact compatibility shape.** Decide whether to migrate existing tool
   artifacts to an explicit `kind: "tool"` on write immediately, or only accept
   both shapes on read until the next major schema bump.
2. **Schedule config ownership.** Decide whether recurring schedule metadata
   belongs in `manifest.json`, a new `schedule.json`, or outside the almanac
   directory in caller-owned CI/cron config.
3. **Notification surface.** Decide whether v0.6 only exits with JSON suitable
   for external notifiers, or includes a small webhook command.
4. **Stale lock policy.** Decide whether explicit stale-lock cleanup belongs in
   v0.6 or should wait until a daemon exists.
5. **LLM-backed question mode.** Keep it outside v0.6 unless refresh work lands
   early and the provider/cost contract is explicit.
