# v0.9 RC Smoke

This smoke validates v0.9 ask suite operations: fixture authoring,
suite-level gates, refresh-integrated ask validation, and hardened
doctor/profile readiness. Use fresh disposable roots. Deterministic replay,
fixture authoring, ask-suite, and refresh ask validation do not call an LLM
provider. The real-provider ask section requires `ANTHROPIC_API_KEY`.

Run from the repository root.

## 1. Static Gates

```bash
git switch main
git pull --ff-only

bun run typecheck
bun test
```

## 2. sqlite-demo Ask Suite Operations

Create a fresh offline demo:

```bash
SQLITE_ROOT="$(mktemp -d -t almanac-v090-sqlite-smoke)"

bun src/cli.ts demo --root "$SQLITE_ROOT"
bun src/cli.ts inspect sqlite-demo --root "$SQLITE_ROOT"
bun src/cli.ts profile sqlite-demo --root "$SQLITE_ROOT"
bun src/cli.ts benchmark sqlite-demo --root "$SQLITE_ROOT"
```

Run a deterministic mock-provider answer and save the answer artifact. Keep the
mock responses explicit so the smoke is repeatable:

```bash
export ALMANAC_LLM=mock
export ALMANAC_MOCK_RESPONSES="$(
  cat <<'JSON'
{
  "answer-planner@planner-v1": [
    "{\"action\":\"call_tool\",\"toolName\":\"query_facts\",\"input\":{\"q\":\"transactions atomic\",\"limit\":3}}",
    "{\"action\":\"stop\",\"reason\":\"enough-evidence\"}"
  ],
  "answer-synthesis@synthesis-v1": "{\"status\":\"ok\",\"answer\":\"SQLite transactions are atomic.\",\"citations\":[{\"sourceId\":\"sqlite-transactions\",\"url\":\"https://www.sqlite.org/lang_transaction.html\",\"fetchedAt\":\"2026-01-01T00:00:00.000Z\"}]}"
}
JSON
)"

bun src/cli.ts ask sqlite-demo \
  "Are SQLite transactions atomic?" \
  --save \
  --label rc-answer \
  --json \
  --root "$SQLITE_ROOT"
```

Inspect and replay the saved answer without a provider:

```bash
answer_id="$(
  bun src/cli.ts runs sqlite-demo --kind answer --latest --json --root "$SQLITE_ROOT" \
    | jq -r '.runs[0].runId'
)"

bun src/cli.ts runs sqlite-demo "$answer_id" --json --root "$SQLITE_ROOT"
bun src/cli.ts runs sqlite-demo "$answer_id" --root "$SQLITE_ROOT"

unset ALMANAC_LLM
unset ALMANAC_MOCK_RESPONSES

bun src/cli.ts ask-replay sqlite-demo \
  --from-runs \
  --label rc-answer \
  --json \
  --root "$SQLITE_ROOT"
```

Author a fixture from the saved answer and run the suite gate:

```bash
bun src/cli.ts ask-fixtures init sqlite-demo \
  --json \
  --root "$SQLITE_ROOT"

bun src/cli.ts ask-fixtures add-from-run sqlite-demo "$answer_id" \
  --fixture-id sqlite-transactions-atomic \
  --json \
  --root "$SQLITE_ROOT"

bun src/cli.ts ask-suite sqlite-demo \
  --json \
  --root "$SQLITE_ROOT"
```

Persist refresh-integrated ask validation evidence:

```bash
bun src/cli.ts refresh run sqlite-demo \
  --from-stage 12-benchmark-run \
  --ask-suite \
  --save \
  --label rc-sqlite-ask-suite \
  --json \
  --root "$SQLITE_ROOT"

bun src/cli.ts runs sqlite-demo \
  --kind refresh \
  --latest \
  --json \
  --root "$SQLITE_ROOT"

bun src/cli.ts profile sqlite-demo --root "$SQLITE_ROOT"
bun src/cli.ts doctor sqlite-demo --root "$SQLITE_ROOT"
```

Expected sqlite-demo result:

- benchmark passes,
- saved answer artifact has planner, tool, citation, synthesis, and quality
  trace sections,
- saved-answer replay passes without provider credentials,
- `ask-fixtures add-from-run` creates a reviewable row under `tests/ask.jsonl`,
- `ask-suite` exits `0` and reports one passing fixture,
- saved refresh artifact records `askSuite.status` as `passed`,
- `profile` shows fixture path, latest ask suite status, latest answer quality,
  and `answer mode ready`,
- `doctor` reports the answer readiness check as ok.

## 3. Enterprise AI Compile Smoke

Create a fresh standard-depth Enterprise AI almanac:

```bash
export ALMANAC_ROOT="$(mktemp -d -t almanac-v090-enterprise-ai-smoke)"

bun src/cli.ts new "Enterprise AI" \
  --slug enterprise-ai-smoke \
  --profile mixed \
  --depth standard \
  --scope "Enterprise AI adoption and operations for organizations: AI governance, risk management, evaluation, security, compliance, deployment patterns, observability, and operating-model decisions for production AI systems." \
  --source "https://www.nist.gov/itl/ai-risk-management-framework" \
  --source "https://owasp.org/www-project-top-10-for-large-language-model-applications/" \
  --source "https://atlas.mitre.org/" \
  --root "$ALMANAC_ROOT"

bun src/cli.ts inspect enterprise-ai-smoke --root "$ALMANAC_ROOT"
bun src/cli.ts profile enterprise-ai-smoke --root "$ALMANAC_ROOT"
bun src/cli.ts benchmark enterprise-ai-smoke --root "$ALMANAC_ROOT"

jq -r '.source.sourceId' \
  "$ALMANAC_ROOT/enterprise-ai-smoke/extracted/facts.jsonl" \
  | sort | uniq -c

jq '.results[] | select(.status!="pass") | {fixtureId, kind, reason, observed}' \
  "$ALMANAC_ROOT/enterprise-ai-smoke/.compile/benchmark-result.json"
```

If a provider timeout halts source evaluation, rerun from the failed stage:

```bash
bun src/cli.ts update enterprise-ai-smoke \
  --from-stage 02b-source-discovery-evaluator \
  --no-bump \
  --root "$ALMANAC_ROOT"
```

Expected Enterprise AI compile result:

- all compile stages complete,
- generated benchmark coverage is at or above 8 positive / 5 negative / 13
  total,
- benchmark passes with cited positives and negative refusals,
- `profile` and `doctor` expose answer readiness without hidden provider calls,
- answer readiness may remain `needs-validation` until Enterprise AI answer
  fixtures and suite evidence are created.

## 4. Real Provider Ask And Replay

Run only when `ANTHROPIC_API_KEY` is available:

```bash
test -n "$ANTHROPIC_API_KEY" || {
  echo "ANTHROPIC_API_KEY is required for real provider ask smoke"
  exit 1
}

enterprise_answer_json="$ALMANAC_ROOT/enterprise-ai-smoke-answer.json"

ask_exit=0
bun src/cli.ts ask enterprise-ai-smoke \
  "What governance checks should precede production rollout?" \
  --save \
  --label rc-real-provider \
  --json \
  --root "$ALMANAC_ROOT" \
  > "$enterprise_answer_json" || ask_exit=$?

if [ "$ask_exit" -gt 1 ]; then
  cat "$enterprise_answer_json"
  exit "$ask_exit"
fi

answer_status="$(jq -r '.status' "$enterprise_answer_json")"
if [ "$answer_status" != "ok" ] && [ "$answer_status" != "abstained" ]; then
  cat "$enterprise_answer_json"
  exit 1
fi

jq '{answerId, status, citations: (.citations | length), toolCalls: (.toolCalls | length), quality: .trace.quality.status}' \
  "$enterprise_answer_json"

enterprise_answer_id="$(jq -r '.answerId' "$enterprise_answer_json")"

bun src/cli.ts runs enterprise-ai-smoke \
  "$enterprise_answer_id" \
  --json \
  --root "$ALMANAC_ROOT"

bun src/cli.ts ask-replay enterprise-ai-smoke \
  --from-runs \
  --label rc-real-provider \
  --json \
  --root "$ALMANAC_ROOT"
```

Promote the answer to an Enterprise AI suite fixture only when the saved answer
recorded at least one tool call. A correct abstention with no tool calls should
still pass saved-answer replay, but it cannot become a deterministic fixture
row via `add-from-run`.

```bash
if jq -e '(.toolCalls | length) > 0' "$enterprise_answer_json" >/dev/null; then
  bun src/cli.ts ask-fixtures init enterprise-ai-smoke \
    --json \
    --root "$ALMANAC_ROOT"

  bun src/cli.ts ask-fixtures add-from-run enterprise-ai-smoke "$enterprise_answer_id" \
    --fixture-id enterprise-ai-governance-rollout \
    --json \
    --root "$ALMANAC_ROOT"

  bun src/cli.ts ask-suite enterprise-ai-smoke \
    --json \
    --root "$ALMANAC_ROOT"

  bun src/cli.ts refresh run enterprise-ai-smoke \
    --from-stage 12-benchmark-run \
    --ask-suite \
    --save \
    --label rc-enterprise-ai-ask-suite \
    --json \
    --root "$ALMANAC_ROOT"
else
  echo "saved answer had no tool calls; fixture promotion skipped"
fi

bun src/cli.ts profile enterprise-ai-smoke --root "$ALMANAC_ROOT"
bun src/cli.ts doctor enterprise-ai-smoke --root "$ALMANAC_ROOT"
```

Expected real-provider result:

- answer status is either `ok` with citations or a correct abstention,
- saved answer artifact includes trace and quality sections,
- saved answer replay passes without another provider call,
- if a fixture is promoted, `ask-suite` passes and the saved refresh artifact
  records `askSuite.status` as `passed`,
- latest answer status, latest suite status, and quality gate appear in
  `profile` and `doctor`.
